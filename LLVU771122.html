The LLVU paradigm

A blockchain is a distributed state machine (in short, a DSM) maintained by a set of parties, called nodes. For each state, the DSM specifies a set of valid (or legal) transactions and how the DSM state deterministically evolves as effect of the transactions.

In a permisionless blockchain, anybody can submit a transaction to a node. The valid transactions are ordered into a block and the new state resulting from the ordered execution of the transactions is computed.

The main problem that a DSM must solve is to guarantee that the nodes are in a consensus with respect to the current state. Since the state of the DSM deterministically evolves according to the transactions, it is sufficient for the nodes to reach consensus over the next block of transactions.

Reaching consensus (or agreement) among a set of nodes that might malfunction or that purposedly try to disrupt the consensus process is a classic problem in the theory of distributed system. It is known as the Byzantine Agreement problem, it was first studied by Pease, Shostak and Lamport in the early 1980's, and there is a large corpus of impossibility results, lower bounds and constructions. Unfortunately, none of the protocols proposed was practical, especially for the scale required by a large permissionless blockchain.

A new paradigm arises

The Bitcoin protocol is the first consensus protocol to be executed in a large scale setting. Nodes running the Bitcoin protocol reach consensus on the node that is to propose the next block. In the Bitcoin protocol each node can compute by itself if it is the proposer and, if it is, the node can publish a short proof of this for all the other nodes to verify, followed by the actual new block.

We can summarize the properties of the Bitcoin protocol by introducing the LLVU paradigm:

  1. Local. Every node works in isolation to gain the right to propose the next block.
  2. Laconic. A node only speaks to announce to be the next proposer.
  3. Verifiable. Every node can verify the announcement with no further interaction with the announcer.

In addition to the three properties above, the Bitcoin consensus is also:

  1. Unpredictable. No one can predict who is going to be the next proposer until it reveals itself.

Properties LLV guarantee that, for each new block, the number of messages needed to reach consensus only depends on the number of nodes that can prove to be the next proposer (a quantity that can be controlled by the protocol) and it is independent on the total number of nodes (a quantity that one wants to make as large as possible). Property U is more subtle and it is crucial for security and full distribution, as explained below.

Resiliency to online attacks

It is good practice not to give an attacker well identified target nodes and this is even more challenging if the attack takes place as the protocol is under execution. Not in LLVU! The fact that no one can predict who the next proposer is, until it reveals itself, gives a bad actor no specific target for an offline attack (that is, before the protocol is executed). Moreover, once the proposer manifests itself, by Laconicity, its task is completed. In other words, the target for an online attack appears and it immediately execute its task and disappears before it can be subverted.

“This is the ultimate security,” asserts Micali. "In order to corrupt a powerful user, a bad actor would need to have knowledge about who" is the next proposer.

Fully distributed consensus

A truly distributed protocol should not rely on a node to participate to multiple instances of the protocol. Indeed, how to incentivate nodes to participate is a delicate issue and it becomes more difficult if nodes are required to stick around for multiple executions.
Laconicity implies that the Bitcoin consensus protocol only expects the "victory announcement" message and thus does not rely on any node taking any specific action other than this. In addition, Locality means that cooperation (i.e., message exchanges) is not needed to find out the next proposer. In other words a node can decide when and where to take part and no commitment to multiple rounds or to "coordination" tasks is needed.

Bitcoin's implementation of LLVU

Bitcoin gave the first implementation of the LLVU paradigm by leveraging on the proof of work (PoW) concept. Everytime a new block is announced, a computational puzzle is defined. The puzzle depends on the previous block and different blocks give different puzzles. Every node starts working on solving the puzzle (Locality) and only the node that finds a solution (Laconicity) sends a message including the solution to the puzzle along with the next block. Everybody can verify that the solution is indeed correct (Verifiability) and that all transactions are legal.

There is an inherent tension in the PoW implementation of the paradigm. If the puzzle is too difficult then blocks will take too long to be produced; if the puzzle is too easy then too many nodes will solve it at the same time and they might propose different blocks and the blockchain will fork. Bitcoin makes sure that not too many nodes give a solution by adjusting the puzzle difficulty so to require 10 minutes on average to solve. Nonetheless, forks happen and Bitcoin has a clever way of managing forks by means of economic incentives.

An interval of 10 minutes between successive blocks has proved to be too much for real world applications. In other words, the LLVU implementation provided by Bitcoin has solved the communication complexity of the protocol but at the expenses of delaying the block production

Algorand implementation of LLVU

The Algorand Consensus protocol is a more efficient implementation of the LLVU paradigm that is based on the proof of stake (PoS) concept, by which each node is chosen with probability proportional to its stake (amount on Algo tokens owned).

PoS is efficiently implemented by means of a Verifiable Random Function (a VRF, in short). By simplifying a bit, each node id in Algorand has a personal function

Fid that only the node can evaluate but everybody, given
(x,y), can verify whether
Fid(x)=y. Each node evaluates its own function on the appropriate input and finds out whether it is the next proposer (thus the protocol is laconic). The protocol is such that a node wins if the value of the VRF falls in an interval of size proportional to its stake.
Evaluating the function is certainly a local computation and it is much faster than finding the solution to a Bitcoin puzzle and the function. Moreover, the output of a VRF is verifiable and pseudorandom, in the sense that the value
Fid(x) cannot be guessed better that at random, and thus unpredictable.

The efficiency guaranteed by the LLVU approach as implemented by Algorand is not just a matter of technical pride but it has a direct effect on security. The fact that the number of messages exchanged in the protocol and the computation required by each node are essentially independent on the number of participants makes it possible for everybody to participate. For this reason there is no mimimum stake required to take part to the protocol. The larger the number of participants (and the stake), the more secure the protocol. Algorand is not worried of capping the number of participant. Just like Bitcoin! All are welcome, all contribute.

Unpredictability as fairness and security

Just as LLV have a direct and clear impact on the efficiency of the consensus protocol, understanding the impact of the unpredictability property requires a little more effort.

As discussed above, unpredictability makes online attacks impossible as a bad actor has no clue about which node will play a role and is thus an appropriate target for subversiob. I would like though to also point out that unpredictability is a form of fairness (each contributes according to the stake owned) that carries with itself a subtle security aspect. Being selected to propose the next block is seen differently by a good actor and by a bad actor. A good actor sees the selection as an expensive step and it is happy to contribute if the probability of being selected is at most proportional to its stake. For a bad actor instead acting as a proposer is an opportunity to attack the protocol (for example by trying to inject bad blocks) and would like the probability of being selected to be at least proportional to its stake, so to have an economy of scale for the attacks. Again there are two factors pulling in different directions and the Algorand protocol strikes the right balance.

Breaking this balance could potentially have dangerous consequences. Let us consider a recent proposal from the ETH community, called the Orbit SSF, that goes directly in this directions: nodes with stake above a certain threshold

T are guaranteed to be selected as validators. The objective here is to guarantee a stable core of "validators" and to have a periphery rotating around the core.
As a first observation, this approach explicitly breaks the fully distributed nature of the protocol as it requires nodes to stick around. What is more important though is that nodes with a large stake (including good and bad actors) contribute more than its share. Implicitly, there is an incentive for good actors to stay below the threshold
T. The good actor that, against its best interest, stays above the threshold become preferred targets for attacks. On the other hand, this introduces an economy of scale for bad actors as it provides a finite stake
T that guarantees permanent participation and opportunity to influence the protocol. The benefit of weaking the U dimension must thus be carefully weighted against the potential drawbacks that are introduced.

Conclusions

Bitcoin introduced the LLVU paradigm for consensus thus making consensus, and consequently DSMs, practical. The main benefit of the LLUV are reduced communication and ultimate security from online attacks. In the initial PoW implementation of the LLVU paradigm by Bitcoin, this came at the expense of a computation load that grew with the number of the nodes.

Algorand gives a different implementation of the LLVU paradigm leveraging on PoS and exploiting advanced Cryptography (VRFs) that made communication and computation extremely efficient even for a large number of nodes. Without entering considerations about finality, we can say that Algorand takes less than 3 seconds to produce a block, compared to the 10 minutes needed by Bitcoin.

Algorand strictly adheres to the LLVU paradigm and is thus able to reap all its benefits. Weakening the LLVU paradigm in any dimension has possible drawbacks. Removing any of the LLV properties will introduce more communication from the node. The unpredictability property is more subtle and its weakening has effect on the security of the protocol.
